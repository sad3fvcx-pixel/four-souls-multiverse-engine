# Four Souls Multiverse Engine
# Ability Resolution

Version: 1.0 (Draft)

---

# Назначение

Данный документ описывает полный жизненный цикл способности
от момента возникновения события до полного завершения её выполнения.

Этот алгоритм является обязательным для любой способности,
независимо от типа карты.

---

# Общий алгоритм

Любая способность проходит следующие стадии:

1. Получение события
2. Поиск подходящих способностей
3. Проверка Trigger
4. Проверка Conditions
5. Построение Ability Context
6. Выбор Targets
7. Создание Effect Queue
8. Помещение эффектов в Stack
9. Разрешение Stack
10. Проверка State-Based Actions
11. Генерация новых событий
12. Завершение способности

---

# 1. Получение события

Движок получает событие.

Примеры:

- turn_start
- on_activate
- player_died
- monster_killed
- after_roll

---

# 2. Поиск способностей

Все зарегистрированные способности проверяются.

Способность считается кандидатом,
если её Trigger совпадает с полученным событием.

---

# 3. Проверка условий

Каждое Condition вычисляется.

Если хотя бы одно условие возвращает false,
способность отменяется.

Никаких изменений состояния игры на этом этапе не происходит.

---

# 4. Создание Ability Context

Создаётся новый контекст выполнения.

Контекст содержит:

- источник способности;
- инициатора;
- владельца;
- текущее событие;
- стек локальных переменных;
- результаты предыдущих эффектов.

Контекст существует только во время выполнения способности.

---

# 5. Разрешение целей

Все Target вычисляются.

Если цель не может быть определена,
способность завершается без выполнения эффектов,
если правила карты не требуют иного.

---

# 6. Построение очереди эффектов

Интерпретатор преобразует DSL
в линейную очередь операций.

На этом этапе эффекты ещё не выполняются.

---

# 7. Помещение в Stack

Если эффект является стековым,
создаётся Stack Item.

Если эффект нестековый,
он выполняется немедленно.

---

# 8. Разрешение Stack

Stack работает по принципу LIFO.

Пока стек не пуст:

- взять верхний элемент;
- выполнить эффект;
- сохранить результат;
- удалить элемент.

Во время выполнения могут появляться новые элементы стека.

---

# 9. State-Based Actions

После каждого полностью разрешённого элемента стека
движок обязан проверить игровые состояния.

Например:

- смерть игроков;
- смерть монстров;
- уничтожение предметов;
- получение душ;
- окончание боя.

Все State-Based Actions выполняются
до обработки следующих событий.

---

# 10. Dispatch событий

Если эффект породил новые события,
они помещаются в очередь событий.

Например:

deal_damage

↓

player_died

↓

gain_soul

↓

game_end

---

# 11. Рекурсивное выполнение

Новые события запускают новые способности.

Каждая способность получает собственный Ability Context.

Контексты не разделяют локальные переменные.

---

# 12. Завершение

После окончания выполнения:

- освобождается Ability Context;
- очищаются временные данные;
- управление возвращается игровому циклу.

---

# Ошибки

Если эффект завершился ошибкой:

- Ability Context сохраняется;
- ошибка передаётся обработчику;
- дальнейшее поведение определяется правилами эффекта.

---

# Инварианты

Во время выполнения всегда соблюдаются правила:

- Effects выполняются последовательно.
- Stack разрешается сверху вниз.
- Conditions никогда не изменяют состояние.
- Targets никогда не изменяют состояние.
- Effects являются единственным источником изменений GameState.
- State-Based Actions выполняются после каждого разрешения Stack.
- Все изменения проходят через Event System.
